# Prototype vs. Real Organization: What We're Testing and Why

## What we're testing

Three distinct hypotheses, in priority order:

**1. Can a pre-trained language model separate IT incident types without task-specific training?**

The answer from the prototype is: barely. Base DistilBERT produces within-group similarity of 0.93 and cross-group similarity of 0.92 — a gap of 0.01. That's essentially "no, it can't." All IT-ops text shares vocabulary (`ERROR`, `failed`, `exceeded threshold`), so the model lumps everything together.

**2. Does fine-tuning fix that?**

Yes — dramatically. After fine-tuning on the 450 samples, cross-group similarity drops to -0.04. Different incident types now repel each other in embedding space (137× improvement in the discrimination gap). This is the most important empirical finding from the prototype.

**3. Is temporal correlation + semantic similarity sufficient to group multi-source signals into one incident?**

With base embeddings: no, timestamps do all the work and embeddings add nothing. With fine-tuned embeddings: yes, both signals pull their weight independently.

## Why this specific design

Each design choice maps to a known constraint of the real problem:

| Design choice | Why |
|---|---|
| DistilBERT (not GPT-scale) | CPU-only, < 5 min pipeline, no cloud dependency — matches org IT security constraints |
| Logistic Regression on frozen embeddings | Separation between feature extraction (generic) and classification (task-specific) — lets you swap the backbone without retraining the classifier |
| Rule-based priority, no learned model | Priority keywords are stable and auditable — ops teams need to explain escalation decisions to stakeholders |
| DBSCAN not k-means | Real incident counts are unknown; k-means requires specifying k in advance; DBSCAN auto-discovers clusters and labels noise |
| Temporal gate (60-min window) | Semantic similarity alone is not enough — "JWT expired" from Monday and Friday are semantically identical but operationally unrelated |
| Feedback log | Analyst accept/reject on correlations is the ground truth for eventual retraining |

## What transfers to real organizational data — and what doesn't

### What transfers directly

- The pipeline architecture (ingest → embed → classify → correlate → report) is production-ready as a design pattern
- The temporal + semantic dual constraint is the right approach — real systems use exactly this
- The feedback collection mechanism is exactly how you'd build a human-in-the-loop retraining loop
- The fine-tuning result tells you: if you have even a small labeled dataset (a few hundred examples), fine-tuning pays off enormously

### What the prototype gets wrong about real data

The Limitations section is unusually honest. The three critical gaps are:

**No noise signals.** Every row belongs to an incident group. Real systems have ~80–90% noise (routine health checks, scheduled jobs, benign retries). DBSCAN's `-1` noise label is never exercised here, so false positive clustering rate is completely unknown.

**Synthetic text is too clean and too consistent.** Real ServiceNow tickets have typos, abbreviations, org-specific jargon, and vary enormously in length and formality. Real logs have stack traces, UUIDs, and ephemeral hostnames that carry no semantic signal. The fine-tuned model's 137× discrimination gap was achieved on data generated by the same templates it trained on — real data will be harder.

**Batch-only, no streaming.** The dashboard loads all 450 signals at once. A real SRE workflow requires slot a new incoming signal into existing clusters as it arrives, which needs a different algorithm (nearest-neighbor lookup against a recent-signal store, not a batch DBSCAN rerun).

## What this knowledge specifically enables in a real rollout

- You know fine-tuning is mandatory — base embeddings aren't sufficient. So your data labeling effort (getting ops engineers to label even 300–500 real incidents) is not optional work, it's the critical path.
- You know the temporal window needs to be tuned to your org's incident velocity. A telecom with 100 alerts/minute needs a much tighter window than a company with 10 alerts/day, or you'll get timestamp collisions like the prototype does.
- You know the priority scorer's keyword list will need org-specific terms — "P0", "sev1", internal service names — that aren't in the current list.
- The feedback log schema (accept / reject with reason) is directly reusable as a retraining signal once you have real data accumulating.

The prototype proves the shape of the solution is right. What's unknown is the performance profile on real data — that can only be answered by a pilot on one real incident data source (start with logs, they're most structured) before connecting the others.
